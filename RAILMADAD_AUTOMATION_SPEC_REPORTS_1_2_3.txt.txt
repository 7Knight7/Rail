RAILMADAD AUTOMATION SPECIFICATION
PART 1 — REPORTS 1, 2 AND 3
=================================

PURPOSE
-------

This file is the source of truth for implementing RailMadad automation.

Cursor/Codex must read this file completely before modifying code.

Do not shorten, reorder, simplify, or reinterpret the workflow.

Do not replace explicit steps with generic steps such as:

- apply the filters
- process the data
- merge the reports
- generate output

Every operation described below must be implemented and verified separately.

If the live RailMadad portal differs from this specification, stop and report the exact difference instead of guessing.

The entire workflow must begin only when the user clicks:

"Generate Today's Reports"

from the localhost frontend.


============================================================
SECTION 1 — APPLICATION ENTRY FLOW
============================================================

1. The user starts Chrome with remote debugging enabled.

2. The user manually logs in to RailMadad.

3. The user leaves the authenticated RailMadad MIS tab open.

4. The user opens the localhost application.

5. The user clicks:

   "Generate Today's Reports"

6. The frontend calls the existing backend automation endpoint.

7. The backend calls:

   AutomationService.start()

8. Playwright connects to the existing Chrome instance through CDP.

9. Playwright must not launch another Chrome window.

10. Playwright must not attempt automatic RailMadad login.

11. Playwright must inspect all browser contexts and open tabs.

12. Playwright must identify the authenticated RailMadad MIS tab.

13. Prefer the tab whose URL contains:

    /rmmis/admin/home.jsp

14. Do not select a tab using tab number or tab position.

15. If multiple RailMadad tabs exist, choose the authenticated MIS/admin tab.

16. Bring the selected MIS tab to the foreground.

17. Verify that the user is authenticated by checking one or more of:

    - MIS Reports menu exists
    - Logout button exists
    - report dashboard exists
    - URL is not the login page

18. If the user is not logged in:

    - stop the workflow
    - do not click anything
    - do not navigate
    - do not open another tab
    - return this error code:

      RAILMADAD_NOT_LOGGED_IN

19. The frontend must show:

    Title:
    RailMadad Login Required

    Message:
    Please log in to RailMadad in Chrome before generating reports.

20. After the user logs in and clicks Generate again, the normal workflow must continue.


============================================================
SECTION 2 — GENERAL MIS NAVIGATION RULES
============================================================

1. The left navigation initially may show only top-level menu options.

2. Before opening a numbered report, locate:

   MIS Reports

3. Check whether the numbered MIS Reports submenu is expanded.

4. If the numbered submenu is collapsed:

   - click MIS Reports once
   - wait until the numbered report list becomes visible

5. If it is already expanded:

   - do not click it again
   - continue directly

6. Verify the target numbered report item is visible before clicking.

7. If the report item is below the visible sidebar area:

   - scroll only inside the sidebar
   - do not scroll the entire browser page unnecessarily

8. Do not use hardcoded screen coordinates.

9. Use reliable Playwright locators based on:

   - visible text
   - accessible role
   - id
   - name
   - stable CSS

10. Avoid XPath unless no stable alternative exists.

11. If the submenu does not expand:

    - retry once
    - if it still fails, stop that report with NavigationError

12. After clicking a report item, verify:

    - expected page heading
    - expected MIS report URL
    - expected page form or table exists


============================================================
SECTION 3 — GLOBAL TABLE SORTING RULE
============================================================

This rule applies to every RailMadad report used in this specification.

1. After clicking Submit, wait until the report result table loads.

2. Verify that the result table contains at least one data row.

3. Locate the column header named:

   Received

4. The final sorting order must be descending.

5. To produce descending order, click the Received column header exactly twice.

6. Perform the clicks as follows:

   a. locate the correct Received header

   b. wait until it is visible

   c. wait until it is enabled

   d. click Received once

   e. wait until the table order or sort state changes

   f. click Received a second time

   g. wait until the table becomes stable

7. Do not perform a Playwright double-click event.

8. Perform two separate click actions.

9. Log:

   received_header_found
   received_click_1
   received_click_2
   received_sort_verified

10. Verify descending order numerically.

11. Read the first several non-empty Received values.

12. Confirm that each value is greater than or equal to the next value.

13. Do not continue if descending order is not verified.

14. If verification fails:

    - repeat the full two-click sequence once
    - verify again

15. If it still fails:

    - save a screenshot
    - save page HTML
    - mark that report failed
    - do not silently continue with unsorted data


============================================================
SECTION 4 — GLOBAL HTML TABLE EXTRACTION RULES
============================================================

1. The HTML result table is the source of data.

2. Do not use RailMadad PDF as the ingestion source.

3. Do not use OCR.

4. Do not parse the PDF.

5. Do not use clipboard copy/paste automation.

6. Do not use screenshots as data.

7. After the table loads and sorting completes:

   - read every header
   - read every row
   - read every cell

8. Preserve the displayed column order.

9. Trim leading and trailing whitespace.

10. Collapse repeated spaces for matching purposes.

11. Preserve the original visible cell text for final output.

12. If the table contains pagination:

    - detect all pages
    - extract every page
    - avoid duplicate rows
    - preserve page order

13. If the table uses DataTables pagination:

    - continue until Next is disabled
    - collect all rows

14. Exclude the portal total row from Top-N selection.

15. Keep the total row separately when totals are required.

16. Verify:

    - row count greater than zero
    - column count greater than zero
    - required headers exist

17. Convert the extracted table into a pandas DataFrame.

18. Save raw extracted data inside project storage.

19. Do not save extracted files into Windows Downloads.

============================================================
SECTION 6 — GLOBAL SOUTH CENTRAL RAILWAY HIGHLIGHTING RULE
============================================================

This rule applies to EVERY final report, spreadsheet, table, worksheet, and generated PDF.

1. Inspect every final output row.

2. Search all cells in the row for the text:

   South Central Railway

3. Matching must be:

   - case-insensitive
   - whitespace-normalized
   - valid even if the text appears as part of a longer value

4. If "South Central Railway" appears anywhere in a row:

   - highlight the ENTIRE row
   - use a YELLOW background/fill
   - use BLACK text/font

5. The yellow fill and black text must apply to:

   - every visible cell in the row
   - every hidden cell in the row
   - all merged/output columns belonging to that row

6. Do not highlight only the cell containing "South Central Railway".

7. Do not change only the font colour.

8. The complete row must have:

   Background colour:
   Yellow

   Text colour:
   Black

9. Apply this rule after:

   - merging
   - sorting
   - filtering
   - Top-N selection
   - final row ordering

10. Apply the rule before saving the final Excel file.

11. Preserve the same yellow-row and black-text formatting in the generated PDF.

12. Do not apply this formatting to raw CSV or Parquet files because those formats do not support colours.

13. If the same row contains:

    - hyperlinks
    - numbers
    - percentages
    - blank cells

    retain their values and number formats.

14. Change only:

    - row background to yellow
    - row text to black

15. The highlighting rule must be implemented once as a shared reusable formatting function.

16. Do not duplicate separate South Central Railway formatting code inside each report.

17. Suggested shared function:

    highlight_south_central_railway_rows(worksheet_or_table)

18. The shared function must scan:

    - every used row
    - every used column
    - visible columns
    - hidden columns

19. Log:

    scr_row_detected
    scr_row_highlighted
    scr_highlight_count

20. After formatting, verify that every row containing South Central Railway has:

    - yellow fill across the complete row
    - black text across the complete row

21. If any matching row is only partially highlighted:

    - fail report verification
    - save the generated Excel for debugging
    - report the affected row number
